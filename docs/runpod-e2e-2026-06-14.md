# Runpod E2E cross-host run - June 14, 2026

This document captures the first successful end-to-end cross-host run using:
- source host: local macOS MLX laptop
- target host: disposable Runpod GPU pod
- transport: PermeantOS daemon protocol carried through a temporary Runpod HTTP bridge

## Outcome

Verdict:
- PermeantOS successfully completed a real cross-host migration run.
- The live source side extracted tensors from the local MLX exporter.
- The remote target accepted the capability exchange, verified the encrypted header, streamed all payload blocks, and committed the migration.
- The manifest was captured locally.
- Runpod resources were torn down after completion.

What this proves:
- live source-process extraction over the adapter seam
- cross-host transfer from laptop to cloud target
- target-side header validation and block ingestion
- end-to-end commit flow with a saved benchmark manifest

What this does not yet prove:
- direct registration into a production vLLM runtime's private memory structures
- exact continuation from a fully live target inference runtime

## Environment summary

Source host:
- OS: macOS
- Architecture: aarch64
- Runtime exposure: local MLX exporter on `127.0.0.1:29101`

Target host:
- Provider: Runpod
- Pod image: `runpod/base:1.0.2-ubuntu2204`
- GPU: RTX 3090
- OS: Linux x86_64

Bridge path:
- The Runpod basic SSH proxy allowed shell access but rejected SSH channel forwarding.
- A small HTTP bridge was used to carry the daemon's framed request/response protocol through Runpod's HTTP proxy.

## Operational notes

Important transport note:
- Runpod's proxied SSH was good enough for shell access and setup.
- It was not good enough for SSH port forwarding of the custom daemon protocol.
- The HTTP bridge restored reachability without requiring a new provider or a different cloud image.

Cleanup note:
- The active pod used for the successful run was terminated.
- A post-run API check returned an empty pod list, confirming no active remaining Runpod pods for this test.
